# CD Plan (2026-03-28)

Tài liệu này mô tả kế hoạch làm thêm phần CD cho repo `AI-FOR-EDUCATION` trong bối cảnh hiện tại chưa có server thật. Mục tiêu là chuẩn bị sẵn khung CD ở mức tượng trưng, an toàn và dễ mở rộng, để khi có hạ tầng thật thì chỉ cần thay bước deploy placeholder bằng deploy thực tế.

## 1. Mục tiêu

- Giữ CI và CD tách riêng để dễ quản lý.
- Tạo một pipeline CD mang tính chuẩn bị trước, chưa phụ thuộc server thật.
- Cho phép kiểm tra luồng release cơ bản: build, đóng gói, lưu artifact, ghi nhận môi trường deploy.
- Hạn chế tối đa rủi ro vì hiện tại chưa có hạ tầng production/staging thật.

## 2. Thực trạng hiện tại

- Repo đã có CI cho `frontend`, `backend` và `docker compose` smoke test.
- Chưa có server VPS, VM, cloud app service hay Kubernetes để deploy thật.
- Chưa có domain, reverse proxy, SSL hay secret production đầy đủ.
- Vì vậy phần CD trước mắt nên dừng ở mức placeholder release pipeline.

## 3. Hướng CD phù hợp lúc này

Trong giai đoạn hiện tại, CD nên làm theo hướng "giả lập deploy" thay vì cố triển khai thật.

Phạm vi hợp lý trước mắt:

- Chỉ chạy CD khi merge vào `main` hoặc chạy thủ công bằng `workflow_dispatch`.
- Reuse kết quả CI hoặc build lại ở workflow CD.
- Build image hoặc package cần thiết.
- Lưu artifact để chứng minh bản build sẵn sàng phát hành.
- Tạo log/thông tin release cho môi trường `staging-placeholder` hoặc `production-placeholder`.
- Chưa public service ra Internet.

## 4. Những gì có thể làm ngay

### 4.1 Tạo workflow CD riêng

Đề xuất file:

- `.github/workflows/project-cd.yml`

Trigger nên có:

- `workflow_dispatch`
- `push` vào `main`
- có thể thêm `tags` như `v*` nếu sau này muốn release theo version

## 5. Nội dung workflow CD tượng trưng

Workflow CD phiên bản đầu có thể gồm các job sau:

### 5.1 Job `prepare-release`

Mục tiêu:

- checkout source
- xác định commit SHA, branch, thời gian build
- tạo release metadata

Output gợi ý:

- file `release-metadata.json`
- file `release-notes.md`

### 5.2 Job `build-artifacts`

Mục tiêu:

- build frontend
- đóng gói backend
- chuẩn hóa file môi trường mẫu

Artifact có thể upload:

- frontend build output
- backend source package
- `docker-compose.yml`
- `.env.docker.example`
- tài liệu release metadata

### 5.3 Job `docker-package`

Mục tiêu:

- build Docker images để xác nhận sẵn sàng deploy
- chưa bắt buộc push lên registry nếu chưa có tài khoản hoặc registry chính thức

Tùy chọn tượng trưng:

- chỉ `docker compose build`
- hoặc build image và export thông tin tag giả như `ai-education/frontend:<sha>`

### 5.4 Job `deploy-placeholder`

Mục tiêu:

- không deploy thật
- chỉ ghi nhận rằng bản build đã "sẵn sàng để triển khai"

Cách làm tượng trưng:

- dùng GitHub Environments với tên `staging-placeholder`
- tạo `deployment-summary.md`
- upload artifact deploy summary
- ghi URL placeholder như `not-configured-yet`

### 5.5 Job `notify`

Mục tiêu:

- tổng hợp kết quả CD
- ghi rõ build nào đã sẵn sàng
- nếu lỗi thì báo giống tinh thần CI hiện có

## 6. Kết quả mong muốn sau khi làm bản đầu

Sau khi hoàn thành bản CD tượng trưng, repo sẽ có:

- một workflow CD riêng, tách khỏi CI
- khả năng chạy release pipeline thủ công
- artifact build phục vụ kiểm tra hoặc bàn giao
- metadata rõ ràng cho từng lần "phát hành"
- nền tảng để thay thế bước placeholder bằng deploy thật sau này

## 7. Các secret và cấu hình có thể chuẩn bị trước

Dù chưa có server, vẫn nên chuẩn bị trước danh sách biến và secret để sau này đỡ sửa nhiều:

- `DEPLOY_TARGET` - ví dụ `staging-placeholder`
- `APP_BASE_URL` - tạm thời để trống hoặc `not-configured-yet`
- `DOCKER_IMAGE_PREFIX` - nếu sau này push registry
- `GHCR_PAT` hoặc dùng `GITHUB_TOKEN` nếu đẩy image lên GHCR
- `SSH_HOST`, `SSH_USER`, `SSH_PRIVATE_KEY` - chỉ cần khi đã có server thật

Hiện tại chỉ nên khai báo phần nào thực sự dùng tới, tránh tạo secret thừa.

## 8. Roadmap đề xuất

### Giai đoạn 1 - Placeholder CD

Mục tiêu:

- có workflow CD chạy được
- build artifact thành công
- có deploy summary tượng trưng

Nên làm:

- tạo `.github/workflows/project-cd.yml`
- thêm `workflow_dispatch`
- upload artifact release
- dùng GitHub Environment placeholder

### Giai đoạn 2 - Registry Ready

Mục tiêu:

- sẵn sàng cho việc phát hành image

Nên làm:

- build và tag Docker image theo commit SHA
- cân nhắc push lên `ghcr.io`
- thêm retention policy cho artifact/image

### Giai đoạn 3 - Staging thật

Khi có server đầu tiên, nên ưu tiên:

- VPS Linux hoặc cloud VM đơn giản
- cài Docker và Docker Compose
- deploy bằng SSH + `docker compose pull && docker compose up -d`
- cấu hình `.env` riêng cho server
- healthcheck sau deploy

### Giai đoạn 4 - Production cơ bản

Khi hệ thống ổn định hơn:

- domain thật
- Nginx hoặc Traefik reverse proxy
- HTTPS với Let's Encrypt
- backup MongoDB
- rollback cơ bản

## 9. Cấu trúc CD đề xuất khi có server thật

Luồng đơn giản nên là:

1. CI pass
2. CD build image
3. Push image lên registry
4. SSH vào server
5. Pull image mới
6. Chạy `docker compose up -d`
7. Gọi health endpoint để xác minh
8. Nếu fail thì báo lỗi và giữ log

## 10. Việc chưa nên làm lúc này

Vì chưa có hạ tầng thật, hiện tại chưa cần:

- triển khai production tự động hoàn chỉnh
- cấu hình rollback phức tạp
- blue-green deployment
- Kubernetes
- auto scaling
- monitoring stack đầy đủ

Làm quá sớm các phần này sẽ tăng công bảo trì nhưng chưa tạo ra giá trị thực tế ngay.

## 11. Đề xuất thực tế nhất cho repo này

Nếu muốn làm ngay một phiên bản CD hợp lý, nhẹ và có ích, nên ưu tiên đúng 3 việc:

1. Tạo workflow `.github/workflows/project-cd.yml` chạy thủ công và khi push `main`.
2. Build + upload artifact cho frontend, backend và Docker release bundle.
3. Tạo job `deploy-placeholder` để ghi nhận "release sẵn sàng" thay vì deploy thật.

Đây là mức đầu tư tốt nhất trong giai đoạn chưa có server.

## 12. Khi nào chuyển từ placeholder sang CD thật

Có thể nâng cấp sang CD thật ngay khi đã có tối thiểu các điều kiện sau:

- 1 server chạy Docker ổn định
- 1 file `.env` dành riêng cho môi trường deploy
- 1 cách truy cập an toàn như SSH key
- 1 health endpoint ổn định như `/health`
- xác định rõ deploy lên `staging` hay `production`

## 13. Kết luận

Hiện tại chưa có server nên hướng đúng là làm CD ở mức chuẩn bị trước, không cố deploy giả quá phức tạp. Cách hợp lý nhất là dựng một workflow CD riêng để build, đóng gói, lưu artifact và tạo deploy summary placeholder. Khi có server thật, chỉ cần thay job `deploy-placeholder` bằng job deploy qua SSH hoặc registry là đủ.
