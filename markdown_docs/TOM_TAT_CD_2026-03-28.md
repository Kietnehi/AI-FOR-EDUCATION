# Tóm tắt và review CD (2026-03-28)

Tài liệu này được cập nhật lại theo đúng cấu hình hiện tại trong `/.github/workflows/project-cd.yml`.

Điểm quan trọng nhất: CD hiện tại mới dừng ở mức đóng gói release artifact và tạo báo cáo triển khai giả lập. Workflow chưa publish Docker image lên registry và chưa deploy lên hạ tầng thật.

## 1. Cách CD được kích hoạt

Workflow `CD Dự Án` hiện có 2 cách chạy:

### 1.1 Chạy tay

`workflow_dispatch` với input:

- `target_environment`

Giá trị lựa chọn hiện tại:

- `staging-placeholder`
- `production-placeholder`

### 1.2 Tự chạy sau CI

`workflow_run` khi workflow `CI Dự Án` hoàn tất trên các nhánh:

- `main`
- `kiet`

Với điều kiện:

- job `prepare-release` chỉ chạy nếu `workflow_run.conclusion == 'success'`

Nhận xét:

- Đây là cơ chế CD nối sau CI, nhưng mới ở mức orchestration logic.
- Chưa có bước phát hành ra môi trường thật nên gọi đây là "artifact packaging pipeline" sẽ chính xác hơn "production CD".

## 2. Các biến môi trường mức workflow

Workflow đang thiết lập:

- `SOURCE_SHA`
- `SOURCE_BRANCH`
- `TARGET_ENVIRONMENT`

Ý nghĩa:

- `SOURCE_SHA` và `SOURCE_BRANCH` được lấy từ `workflow_run` nếu CD được kích hoạt sau CI.
- Nếu chạy tay, giá trị quay về `github.sha` và `github.ref_name`.

Nhận xét quan trọng:

- Metadata release có cố gắng gắn với commit nguồn đã kích hoạt CI/CD.
- Tuy nhiên các bước `actions/checkout` trong workflow hiện tại không checkout rõ ràng theo `SOURCE_SHA`.
- Điều này tạo ra rủi ro tài liệu phát hành và artifact có thể không được build từ đúng commit nguồn khi workflow được kích hoạt bằng `workflow_run`.

Đây là rủi ro thật sự của CD hiện tại, không chỉ là vấn đề tài liệu.

## 3. Kiến trúc job hiện tại

CD hiện có 7 job:

1. `prepare-release`
2. `build-frontend-artifact`
3. `package-backend-artifact`
4. `package-docker-bundle`
5. `deploy-placeholder`
6. `release-bundle`
7. `cd-summary`

Ngoài ra có thêm job:

8. `report-failure`

## 4. Job `prepare-release`

Job này tạo metadata phát hành:

- `release/release-metadata.json`
- `release/release-notes.md`

Thông tin được ghi:

- release name
- release version
- commit SHA
- branch
- workflow
- run id
- target environment
- thời gian tạo

Artifact được upload:

- `cd-release-metadata`

Nhận xét:

- Phần này đang làm đúng vai trò chuẩn hóa metadata.
- `release_version` hiện chỉ là dạng placeholder: `placeholder-<sha7>`.

## 5. Job `build-frontend-artifact`

Job này thực hiện:

1. checkout
2. setup Node.js 20
3. `npm ci`
4. `npm run build`
5. đóng gói artifact frontend
6. upload artifact

Artifact đầu ra:

- `frontend-release-bundle-<sha7>`

Nội dung được copy vào release bundle:

- `.next`
- `public`
- `package.json`
- `package-lock.json`
- `next.config.mjs`

Nhận xét:

- Đây là bundle build artifact, không phải bằng chứng rằng frontend đã được deploy.
- Workflow hiện chưa có bước publish image, upload lên object storage hay copy lên server.

## 6. Job `package-backend-artifact`

Job này thực hiện:

1. checkout
2. setup Python 3.11
3. `python -m compileall app`
4. copy source backend vào bundle release
5. upload artifact

Artifact đầu ra:

- `backend-release-bundle-<sha7>`

Nội dung được đóng gói:

- `app`
- `requirements.txt`
- `requirements-test.txt`
- `Dockerfile`
- `pytest.ini`

Nhận xét:

- Job này chỉ kiểm tra syntax compile, không chạy test trong workflow CD.
- Việc đó là hợp lý nếu CD đã được gate bởi CI trước đó, nhưng không thay thế được việc deploy thật.

## 7. Job `package-docker-bundle`

Job này thực hiện:

1. checkout
2. setup Docker Buildx
3. copy `.env.docker.example` thành `.env`
4. `docker compose config > release-docker-compose.resolved.yml`
5. đóng gói file Docker liên quan
6. upload artifact

Artifact đầu ra:

- `docker-release-bundle-<sha7>`

Nội dung đang được đóng gói:

- `docker-compose.yml`
- `.env.docker.example`
- `docker-compose.resolved.yml`

Nhận xét:

- Job này chỉ resolve cấu hình compose, không build và không push image.
- Tài liệu cũ ghi rằng CD publish image lên `ghcr.io` là sai với workflow thực tế.

## 8. Job `deploy-placeholder`

Đây là job quan trọng cần gọi đúng tên.

Nó không deploy thật. Nó chỉ:

1. tải `cd-release-metadata`
2. tạo `release/deploy/deployment-summary.md`
3. ghi thông tin bản phát hành vào GitHub Step Summary
4. upload artifact `deployment-summary-<sha7>`

Nội dung summary hiện xác nhận rõ:

- frontend đã build xong và upload bundle
- backend đã kiểm tra cú pháp và upload bundle
- docker compose đã resolve thành công
- chưa thực hiện triển khai lên server thật

Nhận xét:

- Tên job đã nói đúng bản chất: placeholder.
- Mọi tài liệu diễn giải nó như deploy thật đều sai.

## 9. Job `release-bundle`

Job này tải lại toàn bộ artifact đã tạo:

- metadata
- frontend bundle
- backend bundle
- docker bundle
- deployment summary

Sau đó gộp lại thành:

- `full-release-bundle-<sha7>`

Ý nghĩa:

- tiện cho việc tải một gói đầy đủ để kiểm tra, bàn giao, hoặc làm input cho bước deploy thủ công về sau

## 10. Job `cd-summary`

Job này dùng `actions/github-script` để tạo summary tổng hợp cho run CD:

- trạng thái từng job
- release name
- release version
- target environment
- danh sách artifact chính

Nhận xét:

- Đây là lớp báo cáo tốt cho người vận hành.
- Tuy nhiên summary hiện cũng đang ghi chú rõ đây chỉ là bước phát hành tượng trưng.

## 11. Job `report-failure`

Job này chạy nếu bất kỳ job chính nào của CD fail.

Hành vi:

- tạo mới hoặc cập nhật GitHub Issue
- liệt kê các job fail
- gắn link tới run GitHub Actions

Nhận xét:

- Cơ chế alert ở mức issue đã có.
- Nhưng chưa có tích hợp thêm Slack, email, Teams hoặc webhook ngoài.

## 12. Những gì CD hiện có và chưa có

### 12.1 Đang có

- trigger thủ công
- trigger tự động sau CI thành công
- metadata release
- build artifact frontend
- package artifact backend
- resolve Docker Compose config
- deployment summary dạng placeholder
- full release bundle
- summary trong GitHub Actions
- issue tự động khi CD fail

### 12.2 Chưa có

- login registry
- build và push Docker image
- deploy lên VPS, VM, Kubernetes, ECS hoặc dịch vụ tương tự
- migration step
- secret/runtime config cho môi trường thật
- smoke test sau deploy thật
- rollback strategy
- approval gate theo environment protection

## 13. Kết luận review CD

CD hiện tại chưa phải là pipeline triển khai hoàn chỉnh. Nó là pipeline chuẩn bị phát hành và tổng hợp artifact.

Hai điểm tài liệu cũ sai lệch lớn nhất là:

1. CD chưa publish ảnh Docker lên `ghcr.io`.
2. CD chưa triển khai lên server thật; job `deploy-placeholder` chỉ tạo báo cáo mô phỏng.

Ngoài ra còn một rủi ro kỹ thuật cần ưu tiên xử lý nếu muốn dùng workflow này nghiêm túc:

1. các bước checkout chưa pin rõ `ref` về `SOURCE_SHA`, nên artifact của workflow chạy từ `workflow_run` chưa được đảm bảo build từ đúng commit nguồn

Nếu cần nâng CD lên mức deploy thật, thứ tự hợp lý là:

1. sửa checkout để bám đúng `SOURCE_SHA`
2. thêm build/push image hoặc cơ chế phát hành artifact chuẩn
3. thay `deploy-placeholder` bằng bước deploy thật có secret và health check sau deploy
